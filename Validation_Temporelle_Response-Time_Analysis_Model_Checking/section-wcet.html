Analyse WCET

Synthèse

  • WCET (Worst-Case Execution Time) — borne supérieure garantie du temps d'exécution d'un programme sur un matériel donné ; le BCET est la borne inférieure.
  • 4 facteurs clés — instructions du programme, vitesse du CPU, état interne (pipelines), état du cache mémoire (1–2 cycles vs 3–10 vs >10 000 cycles).
  • Analyse statique — sûre (jamais optimiste) mais pessimiste ; nécessite un modèle précis de l'architecture ; outils : OTAWA, aiT, Heptane.
  • Analyse dynamique — mesures réelles mais non sûre (couverture incomplète) ; variantes : mixte avec couverture, WCET probabiliste.
  • OTAWA — framework open-source (IRIT) : décode le binaire, construit le CFG, abstrait le matériel, calcule les bornes cycle par bloc de base.

Facteurs influençant le temps d'exécution

Analogie : Comme un trajet en voiture, le temps d'exécution dépend de la route choisie (le chemin dans le code), du véhicule (le processeur), et du trafic (état du cache, charge système). Même trajet, temps différent selon les conditions.

Le temps d'exécution d'un programme n'est pas une constante : il varie selon quatre dimensions indépendantes.

  • Instructions élémentaires — la séquence d'instructions exécutées dépend des données d'entrée (branches conditionnelles, boucles). Différents chemins d'exécution → différentes durées.
  • Processeur et sa fréquence — sur un microcontrôleur simple, chaque instruction prend effectivement 1 cycle : fréquence 2× = exécution 2× plus rapide. Sur un microprocesseur moderne, ce modèle naïf est totalement faux.
  • État interne du processeur — les pipelines permettent l'exécution simultanée de plusieurs instructions. Un pipeline stall (dépendance de données, branchement mal prédit) interrompt ce parallélisme et allonge le temps.
  • État de la mémoire cache — l'accès à une instruction ou à une donnée peut prendre des ordres de grandeur très différents selon qu'elle est en cache ou non.
1–2 cycles · Cache L1/L2
3–10 cycles · RAM (DRAM)
>10 000 cycles · Swap (disque)
Attention — idée reçue : "Doubler la fréquence du CPU divise par deux le temps d'exécution." Ceci est vrai sur un microcontrôleur ancien (1 instruction = 1 cycle) mais totalement faux sur un microprocesseur moderne avec caches et pipelines. Les accès mémoire hors cache ne s'accélèrent pas proportionnellement à la fréquence.
Définitions :
WCET Worst-Case Execution Time — borne supérieure du temps d'exécution : le programme ne peut pas prendre plus longtemps que cette valeur sur le matériel considéré.

BCET Best-Case Execution Time — borne inférieure : le programme ne peut pas terminer plus tôt que cette valeur.

Ces bornes dépendent du programme et du matériel cible — un même binaire a un WCET différent sur un ARM Cortex-M4 et sur un x86.

Méthodes d'analyse du WCET

Il existe deux grandes familles de méthodes, chacune avec ses compromis entre sûreté (garantie que le WCET n'est pas sous-estimé) et précision (absence de pessimisme excessif).

Analyse statique Sûre
Analyse le code binaire sans l'exécuter. Construit un modèle abstrait du programme et du matériel. Calcule une borne garantie supérieure.

Outils : OTAWA, aiT, RapiTime, Heptane
Analyse dynamique Non sûre
Instrumente le code et mesure de nombreuses exécutions réelles. Produit une distribution empirique des temps d'exécution.

Outils : compteurs matériels (PAPI), analyseurs logiques, clock_gettime

Garantie mathématique : le WCET calculé est une borne supérieure réelle. Fonctionne sans exécuter le programme. Détecte le pire chemin même s'il est rarement emprunté.

Nécessite un modèle précis de l'architecture (souvent indisponible pour les CPU modernes). Pessimiste : toute inconnue sur l'état du cache ou du pipeline est modélisée de façon conservative → surestimation parfois importante.

Applicable à n'importe quelle architecture sans modèle formel. Mesures précises sur le vrai matériel cible. Facile à mettre en œuvre avec des outils standard.

Non sûre : il est impossible de couvrir tous les chemins et états internes possibles. Le pire cas réel peut ne jamais être mesuré. Ne convient pas pour les systèmes critiques (avionique, médical).

Solutions intermédiaires :
  • Mixte avec couverture — combine mesures dynamiques et analyse de couverture de code pour borner le risque de manquer le pire chemin.
  • WCET probabiliste (pWCET) — utilise la théorie des valeurs extrêmes pour extrapoler une distribution cumulative (CDF) et donner un WCET avec une probabilité de dépassement donnée (ex : $10^{-9}$ par heure pour l'avionique).
Notion de sûreté : Une analyse est sûre si elle garantit que le WCET calculé est une borne supérieure du temps d'exécution réel — c'est-à-dire qu'elle ne sous-estime jamais. L'analyse statique est sûre mais pessimiste ; l'analyse dynamique est précise mais non sûre. Pour les systèmes temps-réel critiques, seule l'analyse statique est acceptable.

Analyse statique avec OTAWA

OTAWA (Open Tool for Adaptive WCET Analysis) est un framework d'analyse statique développé à l'IRIT (Institut de Recherche en Informatique de Toulouse). Il est open-source et extensible via des plugins.

Principe de fonctionnement d'OTAWA :
  1. Décode le binaire du programme (ELF ARM) — pas le source C.
  2. Construit le CFG (Control Flow Graph — graphe de flot de contrôle) : graphe orienté où chaque nœud est un bloc de base (séquence d'instructions sans branchement), et les arêtes représentent les branchements possibles.
  3. Abstrait finement le matériel : modèle du pipeline, politiques de cache, latences des instructions.
  4. Combine CFG + modèle matériel pour borner le nombre d'exécutions de chaque bloc de base et le temps (nombre de cycles) de chacun → WCET = chemin critique dans le CFG pondéré.
Cible matérielle du TD : Microcontrôleur ARM LPC2138 (NXP). Architecture ARMv4, processeur simple sans cache de données, pipeline 3 étages. Ce modèle est directement disponible dans OTAWA (-s lpc2138), ce qui rend l'analyse précise pour cette cible.

Préparation de l'environnement

OTAWA et la chaîne de compilation croisée ARM sont fournis dans une image Docker. Les commandes de démarrage diffèrent selon l'OS hôte.

# Sous Linux $ xhost +local: $ docker run -it --rm -e DISPLAY=:0.0 --net=host registry-ia.ensma.fr/svtr /bin/bash # Sous Windows avec WSL2 $ docker run -it --rm \ -v /tmp/.X11-unix:/tmp/.X11-unix \ -e DISPLAY -e WAYLAND_DISPLAY \ -e XDG_RUNTIME_DIR -e PULSE_SERVER \ --net=host registry-ia.ensma.fr/svtr /bin/bash

Utilisation d'OTAWA — étapes

  • Visualiser le programme source — comprendre la structure de binary_search dans bs.c (tableau de 15 éléments, recherche dichotomique).
    cd wcet ; cat bs.c
  • Compilation croisée ARM — produire un binaire ELF ARM sans libc, avec symboles de debug (-g3), sans protection de pile, pour architecture ARMv4.
    arm-linux-gnueabi-gcc -g3 -fno-stack-protector -marm -march=armv4 \
    -o bs.elf bs.c -nostartfiles -static -T link-flash.ld
  • Premier lancement d'OTAWA — OTAWA signale qu'il ne connaît pas la borne de boucle de binary_search et ne peut pas conclure.
    owcet -s lpc2138 bs.elf
  • Déterminer la borne de boucle N — pour un tableau de 15 éléments, la recherche dichotomique fait au plus $\lceil \log_2(15+1) \rceil = 4$ itérations. Générer et éditer le fichier flowfact :
    mkff bs.elf > bs.ff
# Remplacer '?' par 4 dans bs.ff
  • Lancer owcet avec les flow facts — obtenir le WCET en cycles pour le microcontrôleur LPC2138.
    owcet -s lpc2138 bs.elf -f bs.ff
  • Vérification automatique avec orange — l'outil orange d'OTAWA peut déterminer automatiquement les bornes de boucle par analyse statique du source C.
    orange bs.c main -o bs.ffx
  • Génération des statistiques et visualisation du CFG — obviews.py affiche le CFG annoté avec les contributions WCET de chaque bloc de base.
    owcet -s lpc2138 bs.elf --stats -S
obviews.py bs main
Calcul de la borne de boucle N : Pour un tableau de $n$ éléments, la recherche dichotomique divise l'intervalle de recherche par 2 à chaque itération. Le nombre maximal d'itérations est $N = \lceil \log_2(n+1) \rceil$. Pour $n = 15$ : $\log_2(16) = 4$ itérations exactement. C'est la valeur à inscrire dans le fichier bs.ff.
Outil orange : Orange effectue une analyse de valeurs sur le code C source pour déterminer automatiquement les bornes de boucle. C'est utile en pratique, mais il peut échouer sur des bornes dépendant des données d'entrée. Dans ce cas, l'annotation manuelle dans le fichier .ff reste nécessaire.
Impact de l'optimisation compilateur : Recompiler avec -O1 puis relancer owcet produit un WCET différent — généralement inférieur, car le compilateur supprime des instructions redondantes, simplifie des boucles, et améliore l'utilisation du pipeline. Cela illustre que le WCET est une propriété du binaire, pas du source C.

Analyse dynamique

Quand le modèle de l'architecture cible n'est pas disponible pour un analyseur statique (cas fréquent pour les CPU x86/ARM modernes avec caches multiniveaux et prédiction de branchement), on se rabat sur l'analyse dynamique : instrumenter le code et mesurer de nombreuses exécutions.

Limite fondamentale : L'analyse dynamique est non sûre. Il est mathématiquement impossible de garantir qu'on a observé le pire cas — le chemin le plus lent peut nécessiter une combinaison très rare de données d'entrée et d'état de cache. Une mesure dynamique donne une estimation, jamais une borne garantie.

Variations observées lors des mesures

En compilant et exécutant bs.c plusieurs fois sur le même PC, les temps mesurés varient. Plusieurs phénomènes expliquent ces variations :

À la première exécution, le code et les données ne sont pas en cache (cold start). Les accès suivants bénéficient du cache chaud → exécutions plus rapides.

L'état du pipeline et du prédicteur de branchements varie entre exécutions. Un branchement mal prédit coûte plusieurs cycles de pénalité.

Sur un OS à usage général (Linux, Windows), d'autres processus peuvent préempter l'exécution. Des interruptions matérielles (timer, réseau) ajoutent des latences non déterministes.

Effet de la taille de la boucle sur la moyenne

Observation : En passant l'indice de fin de boucle for de 10 à 100, puis à 1000, la durée moyenne par itération diminue. Explication : pour les petites valeurs (10 itérations), le coût fixe de démarrage (chargement du code en cache, initialisation du pipeline) représente une fraction importante du temps total. Pour 1000 itérations, ce coût est amorti sur toutes les itérations → la durée moyenne par itération se rapproche du régime permanent (cache chaud, pipeline stable).
Bonnes pratiques pour l'analyse dynamique :
  • Effectuer un warm-up : exécuter le code une fois avant de mesurer pour chauffer le cache.
  • Répéter les mesures (plusieurs centaines d'exécutions) et conserver le maximum observé — pas la moyenne.
  • Utiliser des compteurs matériels (PAPI, perf) pour mesurer directement les cycles CPU, plus précis que les horloges logicielles.
  • Désactiver les optimisations dynamiques du CPU (Turbo Boost, C-states) pour réduire la variabilité.
